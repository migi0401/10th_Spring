- PK, FK란?
    - PK
        - PK란 Primary Key 기본 키로 테이블 내에서 데이터를 식별하는 역할을 한다.
        - 주로 “테이블명_id” 이런 식으로 네이밍한다.
    - FK
        - FK란 Foreign Key 외래 키로 각 테이블 내에 있는 데이터들끼리 연관관계를 가질 수 있도록 한다.
        - 주로 “다대다” 연관관계를 구현할 때 외래 키를 이용하여 데이터를 구분한다.

- ERD란?
    - [정의] ER: Entity - Relationship ⇒ 개체 - 관계
      ERD:  Entity - Relationship Diagram 을 뜻한다.
        - “개체” 라고 하면 우리가 일상에서 구분할 수 있는 모든 것을 뜻하고
          ”관계” 는 개체와 개체간의 연관성이다.

    - 주로 와이어프레임 이후에 작성하고, “구현”을 목적으로 설계한다.
      자주 바뀔 수 있기 때문에 완벽한 설계에 몰두하지 않는 것이 좋다!

- 연관관계란? 그리고 연관관계를 설정하는 방법은?
    - 연관관계 : 개체와 개체간의 관계이다.
        - 보통 카디널리티로 설명을 하지만, 엄밀히 말하면 다르다.
            - 연관관계: 두 개체 사이의 연관성이 있다는 사실 그 자체
            - 카디널리티: 구체적으로 “몇 대 몇” 의 관계인지를 나타내는 수치
    - 설정
        - 테이블 간의 외래 키(FK)를 두어 연관관계를 설정할 수 있다.

- 정규화란?
    - 정규화: ERD내에서 중복 데이터를 찾아 제거하는 과정이다.
        - 제 1 정규화: 같은 성격과 내용의 컬럼이 연속적으로 나타나는 컬럼이 존재할 때
          해당 컬럼을 제거하고 새로운 테이블을 생성하여 기존의 테이블과
          1 : N 의 관계를 형성하는 것이다.
            - **“하나의 칸에는 하나의 값만”**

        - 제 2 정규화: 기본 키(PK)가 2개 이상의 컬럼으로 구성된 복합키인 경우에 정규화의 대상이 된다.
          **부분 함수 종속을 제거하는 과정이다.**
            - 학번(PK) - 이름 / 과목코드(PK) - 이름
              → 이름은 과목코드와 상관없이 학번에만 종속된다.
              이런 경우가 “부분 함수 종속”이고,
              제 2 정규화를 통해 “이름” 테이블을 새로 생성해서 해결한다

        - 제 3 정규화: 기본 키(PK)에 의존하지 않고 **일반 컬럼에 의존하는 컬럼들을 분리**하는 과정이다.
          이전적 함수 종속을 제거하는 과정이다.
            - PK에 의존하지 않는 컬럼들을 분리하여 새 테이블을 생성하여 해결한다.

- 반 정규화란?
    - 반 정규화: 개발적 측면이나 성능을 위해서 정규화 과정을 되돌아가는 작업
        - JOIN 줄이기
            - 제 1 정규화를 통해 나눴던 걸 성능과 편의를 위해서 역으로 되돌아간다.
            - 공통적으로 필요한 컬럼을 추가한다.
        - 데이터 처리 비용 줄이기 ⇒ group by 부하 줄이기
            - 그룹핑을 통해 컬럼을 묶고 조회할 수 있다. → 이러한 조회 쿼리가 많다면 부하 발생
            - 아예 컬럼을 새로 추가해서 데이터를 준다.
            - 업데이트 관리가 필요하다.
        - 테이블 분리
            - 성능을 위해 컬럼이나 행을 기준으로 분리
                - 컬럼의 용량이 엄청 많을 때, 자주 조회하면 약간의 부하가 올 수 있다.

- DB에서의 상속 관계 표현은 어떻게 하는가?


    - DB에서의 상속 관계: RDBMS는 상속 관계 개념이 없으므로, 직접 구현한다.
        - 필요성
            - is - a 관계 
            ⇒ 특정 엔티티가 다른 엔티티의 특성을 물려받을 때
            - 부모 타입으로 일관된 쿼리 사용 
            ⇒ 여러 엔티티를 공통적인 부모 타입의 엔티티로 관리하는 경우
    - 상속관계 맵핑 전략
        - 조인 전략
            - **각 엔티티를 별도의 테이블로 구성하고, 조인을 통해 데이터를 관리하는 전략**
            - 각각의 엔티티가 개별 테이블로 구성되어 있다.
                - 장점
                    - 설계가 깔끔하다.
                    - 테이블이 정규화 되어있으며, 외래 키 **참조 무결성 제약조건**을 활용할 수 있다.
                    
                    참조 무결성 제약조건: 자식 테이블의 외래 키(FK) 값이 반드시 부모 테이블의 
                    기본 키(PK)로 존재해야 함을 데이터베이스 차원에서 보장하는 규칙
                - 단점
                    - 조회 시 다수의 조인이 필요해 성능이 떨어질 수 있다.
                    - 데이터 저장 시 2번의 INSERT SQL이 호출된다.
                    
        - 단일 테이블 전략
            - **모든 엔티티를 하나의 통합 테이블로 구성하는 방식이다.**
                - 장점
                    - 조인이 필요 없으므로 조회 성능이 빠르고 쿼리가 단순하다.
                - 단점
                    - 자식 엔티티에 맵핑된 컬럼은 null을 허용해야 한다.
                    - 모든 데이터를 하나의 테이블에 저장하기 때문에 테이블이 커질 수 있다.
        
        - 구현 클래스마다 테이블 전략
            - 부모 클래스(예: Item)는 데이터베이스에 아예 테이블로 만들지 않고, 
            **자식 클래스 (Album, Movie, Book)들만 각각 독립적인 테이블로 만드는 전략**입니다.
                - 장점
                    - 서브타입을 명확하게 구분해 처리할 수 있으며, 
                    not null 제약조건을 적용할 수 있다.
                - 단점
                    - 여러 자식 테이블 조회 시, UNION SQL이 필요해 성능이 떨어진다.
                    **만약 ID 가 10인 item을 찾을 때 그것이 Album 인지 Movie 인지 모른다.
                    그래서 UNION SQL을 사용해서 찾게 된다.**
                    - 자식 테이블을 통합하여 쿼리를 작성하는 것이 어렵다.
        
        - 전략 설정 및 적용 방법
            - 부모와 자식 엔티티에 적절한 어노테이션을 사용한다.
            - 부모
                - @Inheritance(strategy=InheritanceType.XXX)를 사용해 상속 전략을 지정한다.
                    - JOINED: 조인 전략
                    - SINGLE_TABLE: 단일 테이블 전략
                    - TABLE_PER_CLASS: 구현 클래스마다 테이블 전략
                - @DiscriminatorColumn(name="DTYPE") 어노테이션을 추가하여, 
                상속 관계를 구분할 수 있는 DTYPE 컬럼을 생성한다.
                ex) DiscriminatorColumn(name=”DTYPE”)
                    - 이 어노테이션이 있어야 DTYPE 컬럼이 생성되며, 기본 이름은 DTYPE이다.
            - 자식
                - @DiscriminatorValue("XXX") 어노테이션을 사용해
                 DTYPE 컬럼에 저장될 값을 지정한다.
                ex) DiscriminatorValue(”Movie”)
                    - 기본값은 자식 엔티티 클래스 이름이다.

- 인덱스란?
    - 인덱스는 데이터베이스 내의 특정 컬럼이나 컬럼들의 조합에 대한 값과 해당 값이 저장된 레코드의 위치를 매핑하여 데이터베이스 쿼리의 성능을 최적화한다.
      **ex) 책에서 원하는 페이지를 순서대로 찾지 않고 목차를 보고 바로 찾아가는 것**
    - 인덱스는 자료구조 기반으로 작동한다.
        - B-Tree (주로 쓰인다.)
            - 균형 잡힌 이진 검색 트리로,
              O(logN)의 시간 복잡도를 가지고 있다.
            - 각 노드 내 데이터들은 항상 정렬된 상태이다.
        - B+Tree
            - B-Tree의 변형된 구조로 더 많은 키를 가질 수 있다.
            - B-Tree와 다르게 Internal 노드와 leaf 노드로 구분되고,
              데이터는 leaf에 검색을 위한 인덱스는 Internal 에 저장된다.
            - 범위 검색이나 순차 검색에 효율적이다.
    - 장점
        - 검색 대상 범위를 줄여 검색 속도를 빠르게 할 수 있다.
        - 중복 데이터 방지
        - WHERE, 절, ORDER BY 절 등이 사용되는 작업이 더욱 효율적으로 처리된다.
    - 단점
        - 인덱스 생성에 따른 추가 공간 필요 ( MYI 파일이 필요하다.)
        - CRUD 작업 시에도 인덱스를 업데이트 해야 한다.
