- RESTful API란?

  REST 아키텍처 스타일을 잘 따르는 API이다.

  ⇒ REST 원칙을 잘 지킨 서비스 API이다.

    - REST API란
        - URI를 통해 자원을 명시하고, HTTP 메서드를 통해
          기본적인 CRUD를 정의하는 아키텍처 스타일
        - 장점:
            - 무상태성을 이용하여 서버를 수평적으로 증가시킬 수 있다.
            - 클라이언트와 서버 간의 의존성이 낮아 독립적으로 진화할 수 있다.
        - 단점:
            - HTTP 프로토콜에 의존한다.
            - 모든 것을 무상태로 설계할 수 있는 것은 아니다. ex) 로그인 유지

- 멱등성이란?

  한 번 호출하든 100번 호출하든 결과가 똑같다는 의미이다.

    - GET: 한 번 조회든 두 번 조회든 같은 결과가 조회됨
    - **POST: 멱등이 아니다. 만약 결제를 하는 작업이면 두 번 호출하면 중복 결제가 된다.**
    - DELETE: 같은 요청을 여러 번 삭제해도 삭제된 결과는 같다.
    - PUT: 결과를 대체한다. 같은 요청을 여러 번 해도 최종 결과는 같다.
        - 활용
            - 자동 복구
            - 서버가 시간 만료로 응답을 못 주었을 때, 클라이언트에서 같은 요청이 들어온다면
              다시 그 작업을 수행해도 되는지에 대한 판단 근거가 된다.
    - 주의: 멱등성은 중간에 외부 요인으로 인해 리소스가 변경되는 것은 고려하지 않는 개념이다.
        - GET => 중간에 PUT으로 바꾸고 => GET 멱등하지 않다고 판단

- HTTP 메서드 종류

  **HTTP메서드는 리소스에 대한 행위를 나타낸다.**

  ⇒ 클라이언트가 서버에 요청할 때 기대하는 것

    - **GET**
        - 서버에 전달해서 사용하고 싶은 데이터는 query를 통해서 전달
        - 메시지 바디를 사용해서 데이터를 전달할 수 있지만, 많이 사용하지 않는다.
    - **POST**
        - 메시지 바디를 통해 서버로 요청 데이터를 전달
        - **서버는 요청 데이터를 처리**
            - 데이터를 처리하는 모든 기능을 다 수행한다.
            - 변경을 넘어서서 프로세스를 처리해야 하는 경우
            - 새로운 리소스가 생성이 안될 수도 있다.
            - POST/orders/{orderId}/start-delivery (컨트롤 URI에도 사용된다.)
        - **새 리소스를 생성하거나 등록하는데 사용한다.**
        - 다른 메서드로 처리하기 애매한 경우
            - JSON으로 조회 데이터를 넘겨야 하는데, GET메서드를 사용하기 어려운 경우
        - **애매하면 POST**
    - **PUT**
        - 리소스를 대체
            - 리소스가 있으면 대체
                - **리소스를 완전히 대체한다**
                - **기존 리소스를 삭제하고 덮어만든다고 생각하기!**
            - 리소스가 없으면 생성
            - 덮어버린다고 생각하자!
        - 클라이언트가 리소스 식별
            - **클라이언트가 리소스 위치를 알고 지정**
                - PUT/members/100 HTTP/1.1 이런 식
    - PATCH
        - 리소스 **부분 변경** PUT과 다르다!
    - DELETE
        - 리소스 제거
            - 내부에서 일어나는 일
            - Hard Delete: 물리적인 삭제
              ⇒ 복구가 어렵다

                <aside>

              DELETE FROM User WHERE userId = ‘아이디’

                </aside>

            - Soft Delete: 복구를 고려한 삭제 / 논리삭제. ex) ‘휴지통’ 이 있는 느낌
              ⇒ 복구는 쉽지만 DB 용량이 커진다

                <aside>

              UPDATE User SET activated = 0 where userId = ‘아이디’

                </aside>
